# 02_整体架构

## 1. 设计原则

Flutter 版本不是独立产品，而是 PWA 的移动原生延伸。整体架构遵循以下原则：

- **共享 Backend**：Flutter 不直接调用 LLM，所有智能能力通过 Backend API 获得。
- **共享 Database Schema**：与 PWA 使用同一套数据模型。
- **离线优先**：核心数据本地缓存，弱网环境下仍可查看与记录。
- **最小化状态**：UI 状态简洁，业务状态集中在 Repository 与 Service。
- **可测试**：业务逻辑与 UI 分离，方便单元测试。

## 3. 页面组织

Flutter 版本的页面组织围绕核心使用场景设计，保持极简：

```
lib/
├── main.dart
├── app.dart
├── router.dart
├── providers.dart
├── screens/
│   ├── home/
│   │   └── home_screen.dart           # Today One Thing 首页
│   ├── person/
│   │   ├── person_list_screen.dart    # 人物列表（非首页）
│   │   └── person_detail_screen.dart  # 人物详情 + Memory + Chat
│   ├── memory/
│   │   └── memory_capture_screen.dart # 记忆捕获入口
│   ├── chat/
│   │   └── ai_chat_screen.dart        # AI Chat
│   └── settings/
│       └── settings_screen.dart       # 设置
├── widgets/
│   ├── today_one_thing_card.dart
│   ├── memory_input_sheet.dart
│   ├── person_avatar.dart
│   └── ...
├── services/
├── repositories/
├── models/
└── utils/
```

**页面设计原则：**

- 首页即 Decision Dashboard，只展示 Today One Thing。
- 人物详情页不是资料页，而是关系上下文 + 记忆流 + AI Chat。
- 记忆捕获优先底部弹窗（Bottom Sheet），而不是独立页面。
- 设置页极简，主要管理通知、隐私、数据。

## 4. Router

使用 **GoRouter** 进行声明式路由管理。

```dart
final router = GoRouter(
  initialLocation: '/',
  routes: [
    GoRoute(path: '/', builder: (context, state) => const HomeScreen()),
    GoRoute(path: '/person/:id', builder: (context, state) => PersonDetailScreen(id: state.pathParameters['id']!)),
    GoRoute(path: '/chat/:personId', builder: (context, state) => AIChatScreen(personId: state.pathParameters['personId']!)),
    GoRoute(path: '/settings', builder: (context, state) => const SettingsScreen()),
  ],
);
```

**路由原则：**

- 深层链接支持：从推送通知可以直接进入 Today One Thing 或 Person Detail。
- 返回栈简洁：避免过深的页面层级。
- 底部导航仅保留：Home / People / Capture / Chat / Settings（未来根据验证调整）。

## 5. 状态管理方案

使用 **Riverpod** 作为主要状态管理方案。

| 层级 | 状态类型 | 工具 |
|------|---------|------|
| **UI 局部状态** | 输入框、弹窗、动画 | StatefulWidget / HookWidget |
| **页面状态** | 列表加载、表单验证 | StateNotifier / AsyncNotifier |
| **全局状态** | 当前用户、设置、通知权限 | Provider / StateNotifierProvider |
| **服务依赖** | Repository、Service 实例 | Provider |

**状态管理原则：**

- 业务逻辑尽量放在 Service/Repository，不在 UI 中处理。
- 异步状态使用 AsyncValue，统一处理 loading/error/data。
- 避免全局状态滥用，仅用户、设置、通知等真正全局的状态使用全局 Provider。

## 6. Repository Pattern

Repository 负责数据的本地与远程抽象。

```dart
abstract class MemoryRepository {
  Future<List<Memory>> getMemoriesForPerson(String personId);
  Future<Memory> createMemory(MemoryInput input);
  Future<void> deleteMemory(String id);
  Future<void> syncMemories();
}

class MemoryRepositoryImpl implements MemoryRepository {
  final LocalMemoryDataSource local;
  final RemoteMemoryDataSource remote;

  MemoryRepositoryImpl({required this.local, required this.remote});

  // 优先读本地，后台同步远程
}
```

**Repository 原则：**

- 每个核心实体一个 Repository：PersonRepository、MemoryRepository、ReminderRepository、AIConversationRepository、SettingsRepository。
- Repository 不直接调用 HTTP，而是调用 Local/Remote DataSource。
- 写操作先写本地，再异步同步到远程，保证离线可用。
- 同步冲突由 Sync Engine 处理。

## 7. Service Layer

Service 负责业务逻辑 orchestration。

| Service | 职责 |
|---------|------|
| **PersonService** | 人物 CRUD、关系上下文管理 |
| **MemoryService** | 记忆捕获、查询、分类 |
| **ReminderService** | Today One Thing 生成、本地通知调度 |
| **AIService** | 调用 Backend API 获取 AI 建议、Chat 回复 |
| **NotificationService** | 本地通知、推送权限管理 |
| **SyncService** | 本地与远程数据同步 |
| **AuthService** | 用户认证、Token 管理 |

**Service 原则：**

- Service 可以组合多个 Repository。
- Service 不持有 UI 状态。
- Service 方法可单元测试。

## 8. AI Layer

Flutter 不直接调用 LLM。AI Layer 是一个轻薄的 Service + Model 层，统一调用 Backend API。

```dart
class AIService {
  final BackendClient client;

  Future<TodayOneThing?> getTodayOneThing();
  Future<List<MemorySuggestion>> suggestMemories(String personId);
  Future<String> chat({required String personId, required String message});
  Future<WhyNowExplanation> explainWhyNow(String personId);
}
```

**AI Layer 原则：**

- 所有 AI 推理在 Backend 完成。
- Flutter 只负责展示 AI 返回的结构化结果。
- AI 响应需要有 fallback：Backend 不可用时的本地默认行为。

## 9. Local Cache

本地缓存用于离线可用与性能。

| 数据 | 存储方式 | 说明 |
|------|---------|------|
| **Person** | SQLite / Drift | 关系型数据，需要查询与索引 |
| **Memory** | SQLite / Drift | 数量可能较大，需要按时间/人物查询 |
| **Reminder** | SQLite / Drift | 需要精确调度 |
| **Settings** | SharedPreferences / Hive | 轻量 KV |
| **AI Conversation** | SQLite / Drift | 聊天记录需要分页加载 |

**Local Cache 原则：**

- 敏感数据本地加密存储。
- 默认优先本地读取，后台刷新远程。
- 定期清理过期缓存，但核心记忆永不删除（除非用户主动）。

## 10. Notification

通知系统分层设计：

| 类型 | 触发方式 | 示例 |
|------|---------|------|
| **Today One Thing** | Backend 计算 + 本地调度 | "今天有一个人需要你" |
| **Birthday Reminder** | 本地日历计算 | "Alice 明天生日" |
| **Follow-up Reminder** | 本地调度 | "你答应 Bob 的方案发了吗？" |
| **Gentle Reminder** | 本地推断 + 用户确认 | "最近很久没有记录，还好吗？" |
| **Remote Push** | Backend 推送 | 未来跨设备同步、重要 AI 发现 |

**Notification 原则：**

- 绝大多数通知由本地生成，不依赖 Backend。
- 通知必须有明确的 Why Now。
- 用户可一键关闭任何通知类型。
- 通知行动按钮直达 Today One Thing 或 Capture。

## 11. Sync

同步策略：

- **双向同步**：本地修改推送到远程，远程修改拉取到本地。
- **冲突解决**：以最后写入为准（Last-Write-Wins），关键操作由用户确认。
- **同步时机：**
  - App 启动时
  - 用户执行写操作后
  - 网络恢复时
  - 后台任务（受系统限制）
- **离线优先**：写操作永远先成功于本地，再异步同步。

## 12. 安全与隐私

- 本地数据加密（SQLCipher / encrypted_shared_preferences）。
- Token 安全存储（Flutter Secure Storage）。
- 不缓存敏感 AI 输出到非加密存储。
- 用户可一键导出/删除所有数据。

## 13. 总结

Flutter 整体架构是一个以 Backend 为智能中枢、以本地缓存为基础、以 Riverpod 为状态骨架、以极简 UI 为表现层的消费级 AI 应用架构。所有设计都服从一个目标：**让 LifeOS 在移动端成为安静、可信、无处不在的关系守护者。**
