# 03_数据库设计

## 1. 核心原则

Flutter 版本与当前 PWA **共用同一套数据库 Schema**。数据模型先由 PWA 验证，Flutter 启动时直接复用已稳定的设计。

设计原则：

- **PWA 先行验证**：所有表结构、字段、索引先在 PWA 中跑通。
- **Flutter 复用**：Flutter 使用同一 Schema，不独立设计数据库。
- **本地 + 远程两层**：Flutter 本地有 SQLite 缓存，远程与 PWA 共享 Supabase/Backend Database。
- **离线优先**：本地数据库是 truth source，远程是同步与备份。

## 2. 数据实体

### 2.1 Person

代表用户关心的一个人。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID / String | 主键 |
| `name` | String | 姓名 |
| `avatar_url` | String? | 头像 URL（可选） |
| `relationship_context` | String? | 关系角色：friend, family, partner, colleague 等 |
| `birthday` | Date? | 生日（可选） |
| `notes` | String? | 简短备注（非 Memory） |
| `importance` | Int | 用户标记的重要程度，影响 Why Now 优先级 |
| `created_at` | DateTime | 创建时间 |
| `updated_at` | DateTime | 更新时间 |
| `sync_status` | Enum | synced / pending / conflict |

**说明：**
- 不存储 MBTI、Big Five、Personality Summary 等描述性标签。
- 只存储对维护关系有实际作用的信息。

### 2.2 Memory

代表关于一个人或一段关系的记忆片段。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID / String | 主键 |
| `person_id` | String | 关联 Person |
| `content` | String | 记忆内容（文本或语音转写） |
| `source_type` | Enum | text / voice / chat_share / screenshot / import |
| `source_uri` | String? | 原始来源标识（如聊天截图本地路径） |
| `meaning_tags` | List<String>? | 意义标签：emotion, promise, event, milestone 等 |
| `captured_at` | DateTime | 捕获时间 |
| `created_at` | DateTime | 创建时间 |
| `confidence` | Int? | AI 对记忆重要性的内部判断（不展示） |
| `sync_status` | Enum | synced / pending / conflict |

**说明：**
- Memory 是 Capture 的核心产物。
- 不存储完整聊天记录，只提取对关系有意义的关键信息。
- `meaning_tags` 用于 Why Now Engine 筛选重要记忆。

### 2.3 Reminder

代表 AI 判断需要在某个时间点提醒用户的事项。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID / String | 主键 |
| `person_id` | String | 关联 Person |
| `type` | Enum | reconnect / follow_up / birthday / support / none |
| `title` | String | 简短标题 |
| `description` | String? | 说明 |
| `suggested_action` | String | 建议的最小行动 |
| `scheduled_at` | DateTime | 计划提醒时间 |
| `status` | Enum | active / done / dismissed / expired |
| `why_now_reason` | String? | 为什么现在（用于建立信任） |
| `created_at` | DateTime | 创建时间 |
| `completed_at` | DateTime? | 完成时间 |
| `sync_status` | Enum | synced / pending / conflict |

**说明：**
- Reminder 是 Today One Thing 的数据来源。
- 一次只展示最高优先级的 active reminder。
- 用户执行/忽略/ dismiss 后更新状态。

### 2.4 Timeline

代表关系演变的时间线事件。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID / String | 主键 |
| `person_id` | String | 关联 Person |
| `type` | Enum | first_note / meaningful_interaction / birthday / milestone / ai_insight 等 |
| `title` | String | 事件标题 |
| `description` | String? | 事件描述 |
| `occurred_at` | DateTime | 事件发生时间 |
| `source_memory_id` | String? | 来源 Memory ID |
| `sync_status` | Enum | synced / pending / conflict |

**说明：**
- Timeline 从 Memory、Reminder、AI 分析中自动生成。
- 不单独由用户维护。
- 用于 AI Chat 和 Why Now 解释。

### 2.5 AI Conversation

代表用户与 AI 关于某个人或某个话题的对话历史。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | UUID / String | 主键 |
| `person_id` | String? | 可选，关联某个人 |
| `role` | Enum | user / assistant |
| `content` | String | 对话内容 |
| `context_snapshot` | JSON? | 对话时的记忆快照（用于可解释性） |
| `created_at` | DateTime | 创建时间 |
| `sync_status` | Enum | synced / pending / conflict |

**说明：**
- AI Chat 基于长期关系上下文。
- 本地保留最近 N 条，完整历史在 Backend。

### 2.6 Settings

用户设置与偏好。

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | String | 固定单条记录 |
| `language` | String | zh / en |
| `theme` | Enum | light / dark / system |
| `notifications_enabled` | Bool | 总通知开关 |
| `today_one_thing_enabled` | Bool | Today One Thing 开关 |
| `birthday_reminder_enabled` | Bool | 生日提醒开关 |
| `weekly_brief_enabled` | Bool | Weekly Brief 开关（默认关闭） |
| `ai_privacy_mode` | Bool | 是否强制本地处理 |
| `sync_enabled` | Bool | 是否开启云同步 |
| `created_at` | DateTime | 创建时间 |
| `updated_at` | DateTime | 更新时间 |

## 4. 索引设计

| 表 | 索引字段 | 用途 |
|----|---------|------|
| Memory | `(person_id, captured_at)` | 按人物查询最近记忆 |
| Memory | `(meaning_tags)` | Why Now Engine 筛选 |
| Reminder | `(person_id, status, scheduled_at)` | 查询 active reminder |
| Reminder | `(status, scheduled_at)` | Today One Thing 查询 |
| Timeline | `(person_id, occurred_at)` | 时间线展示 |
| AI Conversation | `(person_id, created_at)` | 聊天记录分页 |

## 5. 数据同步方案

### 5.1 同步模型

- **Single Source of Truth**：本地数据库。
- **Remote**：Backend / Supabase。
- **Sync Engine**：负责双向同步与冲突解决。

### 5.2 同步流程

```
本地写操作
    ↓
写入 Local DB，标记 sync_status = pending
    ↓
网络可用时，推送到 Remote
    ↓
Remote 确认后，标记 sync_status = synced
```

```
App 启动 / 网络恢复 / 用户下拉刷新
    ↓
拉取 Remote 变更
    ↓
与 Local 合并
    ↓
冲突时按策略解决
```

### 5.3 冲突解决策略

- **默认策略**：Last-Write-Wins（以最后更新时间为准）。
- **关键数据**：Reminder 状态、Settings 等需要用户确认的，触发冲突提示。
- **Never 自动覆盖用户已 dismiss 或已完成的 Reminder。**

### 5.4 离线行为

- 所有读操作优先本地。
- 所有写操作先本地成功，再后台同步。
- 用户离线时创建的数据，上线后自动同步。

### 5.5 数据保留与删除

- 核心记忆（Memory）永不自动删除。
- 用户可一键导出所有数据。
- 用户可一键删除所有数据（GDPR / 隐私合规）。
- 过期 Reminder 可自动归档，但保留供 Timeline 使用。

## 6. 安全与隐私

- 本地数据库加密（SQLCipher）。
- 敏感字段（Memory 内容、AI Conversation）不存储明文日志。
- 同步使用 HTTPS，Token 安全存储。
- 用户可关闭云同步，完全本地使用。

## 7. 与 PWA 的共享策略

| 层级 | PWA | Flutter |
|------|-----|---------|
| Database Schema | ✅ 主导设计 | ✅ 复用 |
| Backend API | ✅ 主导实现 | ✅ 调用 |
| Local Storage | localStorage / IndexedDB | SQLite / Drift |
| Sync Engine | 已实现 | 复用协议 |
| AI Prompt | 已验证 | 复用 |

**Flutter 启动时，PWA 应该已经提供：**

1. 稳定的数据库 Schema。
2. 完整的 Backend API 文档。
3. 经过验证的 AI Prompt 策略。
4. 成熟的 Sync 协议。

## 8. 总结

Flutter 的数据库设计不是独立的，而是 PWA 数据模型的自然延伸。核心实体只有 6 个：Person、Memory、Reminder、Timeline、AI Conversation、Settings。所有设计都围绕一个目标：**让用户的关系记忆安全、持久、可用，并且在正确的时间被温柔地唤醒。**
